### 4.2 핵심 테이블 상세: NIC66B

```sql
CREATE TABLE NIC66B (
  SSPS_DL_CRT_DT  VARCHAR(8)      NOT NULL,  -- 사건생성일자 (YYYYMMDD) [PK1, 파티션키]
  SSPS_DL_ID      VARCHAR(30)     NOT NULL,  -- 사건ID (STR20260308001) [PK2]
  SSPS_TYP_CD     VARCHAR(20),               -- 사건유형코드 (STR001/CTR001 등)

  RPR_PRGRS_CCD   VARCHAR(5),                -- 보고진행상태 ★
                                             --   9  = 제출전(대기)
                                             --   97 = 검토중
                                             --   98 = 완료(승인)
                                             --   10 = FIU제출
  DL_P_RNMCNO     VARCHAR(20),               -- 거래당사자 주민/사업자번호 → NIC01B
  RSK_MRK         DECIMAL(5,2),              -- 위험점수 (분석가 판단, 0~100)
  RSK_GRD_CD      VARCHAR(5),                -- 위험등급 (H/M/L)
  OK_CCD          CHAR(1),                   -- 승인여부 (0=미, 1=승인)
  INST_ID         VARCHAR(50),               -- 기관ID (=결재번호 APP_NO)
  SCNR_MRK        DECIMAL(5,2),              -- 시나리오 점수 (자동 계산)
  AI_RESULT       VARCHAR(100),              -- AI 분석결과
  FIU_RPT_NO      VARCHAR(50),               -- FIU보고번호 (제출 후 발급)
  FIU_RPT_DT      VARCHAR(8),                -- FIU보고일자
  REG_ID          VARCHAR(20),               -- 등록자 → C_USER
  REG_DT          TIMESTAMP,
  UPD_ID          VARCHAR(20),               -- 수정자 → C_USER
```

## 5. AML 도메인 모듈별 상세

### 5.1 STR (의심거래보고)

**정의**: 자금세탁·테러자금조성 의심 거래 → FIU 보고

**관련 테이블**: NIC66B(주), NIC67B(보고내용), NIC70B(당사자), NIC73B(거래), NIC75B(금액), NIC78B(상세), NIC68B(문서), AML_APPR(결재)

**상태 코드 흐름**:
```
RPR_PRGRS_CCD:
  9(대기) → 97(검토중) → 98(완료/승인) → 10(FIU제출)
                 ↓반려
                9(대기)
```

**결재 라인 코드 (GYLJ_LINE_G_C)**:
```
TMS1: STR 1단계 결재 (분석가 → 매니저)
TMS2: STR 2단계 결재 (매니저 → 책임자)
RA1:  RA 1단계 결재
RA2:  RA 2단계 결재
RA3:  RA 3단계 결재
```

**상태코드 (SN_CCD)**:
```
N: 대기 (미결재)
E: 완료 (승인)
R: 반려
```

**STR ID 생성 규칙**: `STR` + `YYYYMMDD` + 4자리 순번 (예: `STR20260308001`)

**보고서 구성**:
```
NIC67B:
  RPR_RSN_CNTNT: 보고사유 (전문 서술, VARCHAR(4000))
  ITEM_CNTNT1~6: 보고항목 최대 6개 (각 VARCHAR(2000))
  DOBT_DL_GRD_CD: 의심거래등급 (H/M/L)
```

### 5.2 CTR (고액현금거래보고)

**정의**: 10백만원 이상 현금 거래 → FIU 자동 보고

**원본과 재구축 차이**: STR과 동일한 NIC66B 테이블 공유
- SSPS_DL_CRT_CCD = 'CTR'로 구분
- SSPS_DL_ID = 'CTR20260308001' 형식

**CTR 흐름 (간소화)**:
```
현금거래 발생 (≥10M) → 자동감지 → NIC66B INSERT (CTR) → FIU 보고
```

### 5.3 KYC (고객확인)

**테이블**: `KYC_CUST`

**주요 필드**:
```
KYC_TP_CCD:   KYC 유형
  INITIAL     신규 고객
  PERIODIC    정기 KYC
  TRIGGERED   이벤트 기반 (거래 패턴 변경 등)
  EDD         강화고객확인 (Enhanced Due Diligence)

KYC_STS_CCD:  KYC 상태
  PENDING     대기
  IN_PROGRESS 진행중
  COMPLETE    완료
  EXPIRED     만료
  REJECTED    거부

PEP_F:        정치적 노출인사 여부 (Y/N)
BENEFICIAL_F: 실제 소유자 여부 (Y/N)
FUND_SRC_CCD: 자금 출처 (SALARY/BUSINESS/INVEST/INHERIT/OTHER)
```

### 5.4 RBA (위험평가)

**RBA 평가 알고리즘** (RiskAssessmentService.java):
```java
// 6개 평가 항목
I001 (COUNTRY):     natCd != 'KR' → 30점; 'KR' → 5점
I002 (OCCUPATION):  FUND_SRC_CCD 기반
                    - OTHER/INHERIT → 20점
                    - BUSINESS → 10점
                    - SALARY → 5점
I003 (TRANSACTION): 고객 기존 riskScr 활용
I004 (PEP):         pepYn='Y' → 30점; 'N' → 0점
I005 (BENEFICIAL):  beneficialYn='Y' → 20점; 'N' → 0점
I006 (CUST_TYPE):   CORP(법인)→15, FORE(외국인)→20, INDV(개인)→5점

최종점수 = Σ(itemScr × itemWght / 100)
EDD 강제 규칙: eddYn='Y' → 등급 강제 'H'

등급 기준:
  H (High):   75점 이상
  M (Medium): 50~74점
  L (Low):    0~49점
```

**테이블 구조**:
```
RA_ITEM (평가항목 정의)
  RA_ITEM_CD: I001~I006
  MAX_SCR:    항목 최대점수
  WGHT:       가중치 (%)

RA_RESULT (평가결과)
  RA_ID (AUTO), RNMCNO, EVAL_DT, RA_GRD, RA_SCR, EDD_F, NEXT_EVAL_DT

RA_RESULT_DTL (항목별 점수)
  RA_DTL_ID (AUTO), RA_ID, RA_ITEM_CD, ITEM_VAL, ITEM_SCR, ITEM_LST_SCR
```

### 5.5 TMS (거래모니터링)

**시나리오 분류 (SCNR_CCD)**:
```
STR: 의심거래 시나리오 (고액출금의심, 구분송금, 제재자거래, 불명의거래 등)
CTR: 현금거래 시나리오 (10M 이상 현금거래)
KYC: 고객확인 시나리오 (고위험강화실사, PEP모니터링)
```

**임계치 구조 (TMS_SET_VAL)**:
```
PK: (SCNR_ID, CS_CCD, SET_VAL_CD)
예:
  SCNR_ID='STR001', CS_CCD='IND', SET_VAL_CD='THRESHOLD_AMOUNT' → SET_VAL='10000000'
  SCNR_ID='STR001', CS_CCD='CORP', SET_VAL_CD='TRANSACTION_COUNT' → SET_VAL='5'
```

**임계치 변경 결재 흐름**:
```
1. 분석가: TMS_SET_VAL_APP INSERT (APP_STS_CCD='N')
2. TMS_SET_VAL_APP_DTL: 변경상세 INSERT
3. AML_APPR 결재 상신
4. 매니저 승인 → AML_APPR.SN_CCD='E'
5. TMS_SET_VAL UPDATE (새 임계치 적용)
6. TMS_SET_VAL_APP.APP_STS_CCD='E'
```

## 11. 업무 프로세스 흐름

### 11.1 STR 처리 전체 흐름
```
[단계 1] 거래 감지 (야간 배치)
  거래 발생 → TMS 시나리오 검사 → 임계치 초과 또는 패턴 탐지
       ↓
[단계 2] ALERT 생성 → 분석가 배정
  분석가(analyst1): 알림 확인, 거래 분석, 고객 위험도 확인
       ↓
[단계 3] 사건(Case) 생성 → NIC66B INSERT
  INSERT NIC66B (RPR_PRGRS_CCD='9')
  INSERT NIC70B (거래당사자)
  INSERT NIC73B (개별 거래 목록)
  INSERT NIC75B (계좌별 금액)
       ↓
[단계 4] STR 보고서 작성 → NIC67B INSERT/UPDATE
  보고사유 작성 (RPR_RSN_CNTNT)
  항목별 상세 작성 (ITEM_CNTNT1~6)
  상태 변경: RPR_PRGRS_CCD='97' (검토중)
       ↓
[단계 5] 결재 상신 → AML_APPR INSERT
  APP_NO 생성 (STR20260308001)
  AML_APPR INSERT (SN_CCD='N')
  NIC66B.INST_ID = APP_NO 업데이트
       ↓
[단계 6] 매니저 결재 → AML_APPR UPDATE
  승인: SN_CCD='E' → NIC66B.RPR_PRGRS_CCD='98'
  반려: SN_CCD='R' → NIC66B.RPR_PRGRS_CCD='9'
       ↓
[단계 7] FIU 제출 → NIC66B UPDATE
  FIU_RPT_NO = 'FIU2026030800001' (FIU 발급)
  FIU_RPT_DT = '20260308'
  RPR_PRGRS_CCD = '10' (FIU제출 완료)
```

### 11.2 WLF 스크리닝 흐름
```
고객 등록/수정
  ↓
배치: LuceneFactivaSearch.search() 실행
  검색어: 고객명 (CS_NM)
  조건: 국적(NTN_CD), 생년월일(DOB)
  임계치: 90F 이상
  ↓
매칭 결과 존재 시:
  INSERT WL_SCREEN_RSLT (MATCH_ST_CCD='PENDING')
  ↓
분석가 수동 검토:
  TRUE_MATCH  → STR 생성 권고
  FALSE_MATCH → 동명이인, 무관
```

### 11.3 RBA 평가 흐름
```
정기 배치 또는 이벤트 트리거
  ↓
RA_ITEM 평가항목 로드 (6개)
  ↓
고객 속성 기반 항목별 점수 계산
  ITEM_SCR = 항목기준점수
  ITEM_LST_SCR = ITEM_SCR × ITEM_WGHT / 100
  ↓
최종점수 = Σ(ITEM_LST_SCR)
  ↓
RA_GRD_STD와 비교 → 등급 결정 (H/M/L)
  ↓
EDD 강제 체크 (EDD_F='Y' → 무조건 H등급)
  ↓
INSERT RA_RESULT, RA_RESULT_DTL
UPDATE NIC01B (RSK_GRD_CD, RSK_SCR)
```
